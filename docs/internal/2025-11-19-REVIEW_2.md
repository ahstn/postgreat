# PostGreat Code Review – 2025-11-19

## Executive Summary
- The crate structure, error handling (`snafu` in library code, `anyhow` only in `main.rs`) and modular analysis layout are solid, but several high-priority heuristics from `docs/1` and `docs/2` are either implemented incorrectly or omitted entirely.
- CLI/docs drift is already confusing users: README examples call `-h`/`-P` while the actual flags are `-H`/`-p` in `src/main.rs:31-53`, so copy/paste usage fails before analysis ever starts (`README.md:37-54`).
- The most critical regressions live in the memory, WAL and planner analyzers—`shared_buffers` math is broken, workload detection always assumes OLTP, and planner guidance ignores the storage-type preconditions spelled out in the docs.
- Table/index analysis is still wired to the older `docs/6` playbook; none of the `docs/2` heuristics (soft-delete partial indexes, BRIN candidates, CREATE INDEX CONCURRENTLY readiness) are surfaced, which means the “Logic Source of Truth” requirement is currently unmet.
- Test coverage stops at a few parsing helpers; no unit or integration tests protect the new heuristics, so regressions like the `shared_buffers` bug ship unnoticed.

## Critical Issues
1. **`shared_buffers` recommendation collapses to 0 MB**
   `analyze_shared_buffers` divides `memory_mb()` (already in megabytes) by `1024^2`, so every recommendation becomes zero and the 25% rule in `docs/1 - Foundation Tuning - Core Config.md:29-39` can never be met. The subsequent variance calculation then divides by zero, so every environment is flagged Critical with a suggested value of `0MB` (`src/analysis/memory.rs:31-58`).

2. **Workload detection is hard-coded to OLTP**
   `is_oltp_workload` always returns `true` when active connections and `max_connections` are below 200 because of the unconditional `true` at `src/analysis/wal.rs:148-166`. That prevents the OLAP path (`15-30min checkpoint_timeout`) from ever triggering, contradicting the workload-specific tuning call-outs in `docs/1:5-63`. Large analytical clusters therefore keep getting 5-minute checkpoint advice, which is the opposite of what the docs prescribe.

3. **WAL analysis ignores half of the documented guidance**
   Section D of `docs/1:148-159` focuses on `max_wal_size` *and* `min_wal_size`, plus inspecting whether forced checkpoints (`checkpoints_req`) are causing “bursty I/O.” The implementation only looks at `max_wal_size` and never reads `pg_stat_bgwriter` to tell whether the current limit is being hit (`src/analysis/wal.rs:23-141`). This misses the key “are checkpoints time-based or size-based?” signal the docs call mandatory, so a user can sit in a bursty state without any warning.

4. **Index analysis is still bound to `docs/6`, not `docs/2`**
   All three index checks (`unused`, `low_selectivity`, `failed index-only`) and their rationale strings cite `docs/6` (`src/analysis/table_index/indexes.rs:238-289`). None of the `docs/2` heuristics—soft-delete partial indexes (`docs/2:34-51`), BRIN opportunities for time-series tables (`docs/2:53-69`), or CREATE INDEX CONCURRENTLY safety checks (`docs/2:16-33`)—are implemented, so the analyzer cannot produce the recommendations the inception prompt requires.

## Refactoring Recommendations
1. **Fix `shared_buffers` math and cap**
   _Before (`src/analysis/memory.rs:31-37`):_
   ```rust
   let recommended_mb = (spec.memory_mb() as f64 * 0.25 / (1024.0 * 1024.0)) as u64;
   let recommended_mb = recommended_mb.min(8192);
   let variance = (current_mb as f64 - recommended_mb as f64).abs() / recommended_mb as f64;
   ```
   _After:_
   ```rust
   let mut recommended_mb = (spec.memory_gb as f64 * 1024.0 * 0.25).round() as u64;
   recommended_mb = recommended_mb.min(8192);
   let variance = (current_mb as f64 - recommended_mb as f64).abs() / recommended_mb as f64;
   ```
   Using `memory_gb` keeps everything in megabytes, preserves the 25 % rule, and ensures the 8 GB cap applies only after the base calculation.

2. **Make workload detection data-driven**
   _Before (`src/analysis/wal.rs:148-166`):_
   ```rust
   if let Some(active) = stats.connection_count {
       if active >= 200 {
           return true;
       }
   }
   if let Some(param) = get_param(params, "max_connections") {
       if let Ok(max_conn) = param.current_value.parse::<usize>() {
           if max_conn >= 200 {
               return true;
           }
       }
   }
   true
   ```
   _After:_
   ```rust
   if let Some(active) = stats.connection_count {
       if active >= 200 {
           return true;
       }
       if active <= 20 {
           return false;
       }
   }
   if let Some(param) = get_param(params, "max_connections") {
       if let Ok(max_conn) = param.current_value.parse::<usize>() {
               if max_conn <= 50 {
                   return false;
               }
               if max_conn >= 200 {
                   return true;
               }
       }
   }
   false
   ```
   Returning `false` for clearly analytical footprints unlocks the OLAP branch and lines up with the OLTP/OLAP distinctions in the docs.

## Missing Test Cases
- **`shared_buffers` 25 % rule and 8 GB cap** – Add unit tests that feed a fake `SystemStats`/`PgConfigParam` pair and assert the suggested value for 16 GB, 64 GB, and 256 GB tiers matches `docs/1:29-39`. This would have caught the current zero-recommendation bug.
- **Workload classification** – Introduce tests for `is_oltp_workload` covering low-connection OLAP databases so that the fallback cannot silently regress to `true` again.
- **WAL burst detection** – Integration-style test (can be fixture-based) that feeds `pg_stat_bgwriter` metrics showing `checkpoints_req` > `checkpoints_timed` to ensure a suggestion fires when bursty I/O is detected.
- **Index heuristics from `docs/2`** – Tests that model a soft-delete table (`is_deleted` flag) and a BRIN-friendly append-only table should verify that the analyzer recommends partial/BRIN indexes per `docs/2:34-69`.
